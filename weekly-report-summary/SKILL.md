# 米家App周报简述生成

从团队周报明细中提取关键业务进展，生成精炼的汇总周报简述。

## 业务领域筛选规则

优先将内容归入以下业务领域；无法归入任何既定领域的内容统一放入"其他"：

| 领域 | 关键词匹配规则 |
|------|---------------|
| **配网** | 配网、批量配网、WiFi配网、蓝牙配网、AP配网、配网超时、添加设备 |
| **家庭** | 家庭、家人、家庭成员、设备分享、分享、成员管理、邀请成员、切换家庭、家人关系、滑板车分享 |
| **账号** | 账号、登录、注册、跨账号、账号安全、账号切换 |
| **OTA** | OTA、固件升级、设备版本升级、配件版本升级、多配件升级、模组升级、更新记录 |
| **App强制升级** | 强制升级、App升级、版本升级（特指App自身） |
| **耗材** | 耗材、组合购买、滤芯、耗材余量 |
| **鸿蒙** | 鸿蒙、HarmonyOS、华为 |
| **消息** | 消息、Push、推送、通知、消息中心、通知中心 |

## 执行流程

### Step 1：获取周报明细数据

用户提供周报明细的飞书链接（通常是飞书电子表格/wiki链接）。

```bash
# 先获取表格基本信息
feishu fetch <url>
```

如果是电子表格，读取完整数据：

```bash
# 读取所有有数据的行（通常数据在 A 列，可能合并单元格到多列）
feishu sheet read <token> "Sheet1!A1:A200"
```

如果输出被截断或数据在多列，可以加宽范围：

```bash
feishu sheet read <token> "Sheet1!A1:T50"
```

### Step 2：解析周报明细结构

周报明细的格式特征：
- 每个单元格代表一个人的一项工作内容
- 格式为 **"TOPn：类别名"** + 换行 + 详细内容
- 也可能是 **"其他"** + 换行 + 多条内容（数字编号列表）
- 多个人的内容分布在不同行
- 每行的 A 列包含主要内容，其余列通常为空

解析步骤：
1. 遍历所有非空单元格
2. 提取每个单元格的纯文本内容（处理富文本格式）
3. 识别类别标题（如"Top1：配网"、"Top2：家庭"、"其他"等）
4. 提取该类别下的详细内容

### Step 3：按业务领域分类筛选

对解析出的每条内容：

1. **检查类别标题**是否包含目标领域关键词
2. **检查详细内容**是否包含目标领域关键词
3. 如果匹配，将该条内容归入对应领域
4. 如果不匹配任何目标领域，归入"其他"（不丢弃）

注意处理：
- 一个TOP条目可能涉及多个领域（如"其他"下可能包含鸿蒙和账号两个话题），需要拆分归类
- "其他"类别中的每个编号条目需要单独判断是否属于目标领域
- 同一领域可能有多个人提交的内容，需要合并

### Step 4：内容精炼与总结

对每个领域收集到的原始内容进行精炼：

**精炼原则：**
- 保留关键事实：做了什么、当前状态、预计时间
- 去除冗余细节：具体的会议过程、沟通细节
- 合并重复信息：多人提到同一事项时合并
- 保持简洁：每个领域通常 1-3 个要点，每个要点 1-2 句话
- 如果有飞书文档链接，保留关键链接

**精炼前后对比示例：**

原始：
> Top2： 配网
> 1. 批量配网：三轮测试中，研发在处理车反馈的case
> 2. 服务端配网错误码优化：暂未推动
> 3. ios 选择WiFi页面精确定位权限bug修复：暂未推动
> 4. iOS配网超时AB实验分析报告这周得对一下

精炼后：
> - 配网：
>   - 批量配网：已进入三轮测试，车相关问题研发处理中。
>   - iOS 配网超时 AB 实验报告待对齐。

### Step 5：生成输出

按以下格式输出周报简述（Markdown 无序列表）：

```markdown
- 配网：
  - 要点1
  - 要点2
- 家庭：要点（如果只有一个要点，直接跟在冒号后）
- 账号：要点
- OTA：要点
- 强制升级：要点
- 耗材：要点
- 鸿蒙：要点
- 消息：要点
- 其他：
  - 无法归入上述领域的要点1
  - 无法归入上述领域的要点2
```

**格式规则：**
- 领域名称加粗或不加粗均可，保持与输出模板一致
- 如果某个领域本周无相关内容，不出现在输出中
- 如果某领域只有一个简短要点，直接跟在冒号后面（不换行）
- 如果某领域有多个要点，用缩进子列表
- 每个要点以句号结尾
- 关键飞书文档链接可保留

### Step 6：用户确认与写入

生成后先展示给用户确认。用户确认后，如果用户提供了输出飞书文档的链接，则写入：

```bash
feishu docx update <output_url> --mode overwrite --force -c "<markdown_content>"
```

如果用户没有提供输出链接，则仅在终端展示结果。

## 注意事项

1. **不要编造内容**：只从周报明细中提取信息，不添加任何原始数据中不存在的内容
2. **不要遗漏**：确保所有匹配目标领域的内容都被提取到
3. **保持客观**：不添加主观判断，忠实反映原始周报中的事实
4. **时间节点**：保留重要的时间节点（如"预计11.6版本"、"下周提测"）
5. **飞书链接**：如果原始内容中有飞书文档链接且与该要点直接相关，保留链接
